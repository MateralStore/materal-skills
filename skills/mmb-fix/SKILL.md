# MMB Bug 修复技能

## 概述

本技能提供 MMB (Materal.MergeBlock) 框架项目的系统性 Bug 修复流程，涵盖从错误分析到验证完成的完整工作流。

## 快速开始

### 触发条件

当出现以下情况时使用此技能：
- 构建失败，输出包含编译错误
- 运行时异常，需要定位和修复
- 代码生成失败或生成结果有误
- 用户明确要求"修复"、"解决报错"等类似表述

### 工作流程概览

```
错误输入 → 错误分类 → 定位问题 → 实施修复 → 代码生成 → 构建验证
```

## 工作流程

### 第一步：错误分析

#### 1.1 收集错误信息

**编译错误**：
```bash
dotnet build <解决方案文件路径>
```

**运行时错误**：
- 检查异常堆栈信息
- 收集错误消息
- 识别异常类型

**代码生成错误**：
```bash
MMB GeneratorCode --ModulePath <模块路径>
```

#### 1.2 错误分类

根据错误信息判断错误类型：

| 错误类型 | 特征 | 处理策略 |
|---------|------|---------|
| **编译错误** | CSxxxx 编号，语法错误、类型不匹配 | 修改源代码 |
| **引用错误** | 找不到类型或命名空间 | 检查 using 语句、项目引用 |
| **实体相关** | 实体属性、关系错误 | 修改实体定义 |
| **服务相关** | 服务方法、接口错误 | 修改服务实现 |
| **控制器相关** | 路由、参数错误 | 修改控制器 |
| **配置错误** | appsettings、依赖注入配置 | 修改配置文件 |

#### 1.3 确定影响模块

识别错误涉及的模块：
- 从错误消息中的文件路径提取模块名
- 使用 Glob 查找模块解决方案文件：`**/*.slnx` 或 `**/*.sln`

### 第二步：定位问题

#### 2.1 定位错误文件

**从编译错误提取**：
- 错误消息包含文件路径和行号
- 示例：`AdminServiceImpl.cs(123,45): error CS0246`

**从运行时异常提取**：
- 堆栈跟踪包含完整的调用链
- 定位到具体的方法和行号

#### 2.2 读取相关文件

使用 Read 工具读取错误文件：
```bash
Read <错误文件完整路径>
```

**同时读取相关联的文件**：
- 实体定义（`Abstractions/Domain/*.cs`）
- 服务接口（`Abstractions/Services/*.cs`）
- 服务实现（`Application/Services/*.cs`）
- 控制器（`Application/Controllers/*.cs`）

### 第三步：实施修复

#### 3.1 修复原则

**MGC 文件夹禁止修改**
- ❌ 不要修改 `MGC/` 文件夹下的任何文件
- ✅ 使用 Partial Class 扩展（`Application/Services/*.cs`）

**修复优先级**：
1. **实体定义问题** → 修改 `Abstractions/Domain/*.cs`
2. **服务接口问题** → 修改 `Abstractions/Services/*.cs`
3. **服务实现问题** → 修改 `Application/Services/*.Custom.cs` 或创建新的 Partial Class
4. **控制器问题** → 修改 `Application/Controllers/*.cs`
5. **配置问题** → 修改配置文件

#### 3.2 常见错误修复模式

**类型/命名空间找不到**
```csharp
// 检查 using 语句
using ZhiTu.Core.Abstractions;
using ZhiTu.Main.Abstractions.DTO;
using ZhiTu.Main.Abstractions.Domain;
```

**实体属性错误**
- 在 `Abstractions/Domain/` 中修改实体定义
- 修复后需要重新生成代码

**服务方法签名不匹配**
- 在 `Abstractions/Services/` 中修改服务接口
- 在 `Application/Services/` 中修改服务实现

**控制器路由错误**
- 检查 `[Route]` 特性
- 检查 HTTP 方法特性（`[HttpGet]`, `[HttpPost]` 等）

#### 3.3 使用 Edit 工具修复

```bash
Edit <文件路径> --old_string "<错误代码>" --new_string "<修复后代码>"
```

### 第四步：代码生成

**修复实体或配置后必须重新生成代码**：

```bash
# 确保 MMB 工具已安装
dotnet tool update -g Materal.MergeBlock.Tools

# 为修复的模块生成代码
MMB GeneratorCode --ModulePath <模块路径>
```

**模块路径查找**：
- 使用 Glob 搜索：`**/*.slnx` 或 `**/*.sln`
- 示例路径：`E:\Project\知图\Server\ZhiTu.Main`

### 第五步：构建验证

#### 5.1 执行构建

```bash
dotnet build <解决方案文件路径>
```

#### 5.2 验证结果

**构建成功**：
- ✅ 无错误输出
- ⚠️ 警告可以接受（记录但不阻止继续）

**构建失败**：
- ❌ 返回第一步，分析新的错误
- 迭代修复直到构建成功

## 相关技能

- **`/mmb-generator`**：代码生成规范
- **`/mmb-build`**：构建验证规范
- **`/mmb-entity-design`**：实体设计规范
- **`/mmb-service-impl`**：服务实现规范
- **`/mmb-exception-handling`**：异常处理规范

## 注意事项

1. **禁止修改 MGC**：所有自定义代码必须在非 MGC 目录下
2. **Partial Class 扩展**：服务实现使用 Partial Class 模式扩展
3. **迭代修复**：构建失败时迭代修复，直到成功
4. **完整验证**：每次修复后必须执行代码生成和构建验证
5. **错误记录**：记录所有错误和修复方案，便于后续参考

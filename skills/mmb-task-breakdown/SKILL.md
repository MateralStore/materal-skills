# MMB 任务拆解技能

## 用途

在 MMB 项目中，功能设计完成后进行任务拆解，将功能模块拆解为可并行开发的独立任务。

## 输入

- 功能设计文档路径（如 `docs/FeatureDesign/XXX.md`）

## 输出

- 任务拆解文档（保存到 `docs/Tasks/XXX-Tasks.md`）

## 工作流程

### 1. 读取功能设计文档

读取功能设计文档，分析其中的功能模块。

### 2. 确定任务列表

每个功能模块对应一个任务：
- 任务编号：T-{Module}-{序号}
- 任务名称：功能模块名称
- 优先级：P0（核心功能如登录）或 P1（常规功能）
- 预计工时：根据复杂度估算（0.5h - 2h）

### 3. 分析任务依赖关系

- 所有任务依赖于实体设计完成
- 任务之间应该相互独立，可并行开发
- 如有特殊依赖，在依赖关系图中标注

### 4. 编写任务拆解文档

## 文档结构

```markdown
# {功能名称} - 任务拆解文档

> 基于功能设计文档：[FeatureDesign/XXX.md](../FeatureDesign/XXX.md)
>
> 所属模块：{ModuleName}
>
> 拆解日期：{日期}

---

## 任务总览

| 任务编号 | 任务名称 | 优先级 | 预计工时 | 状态 |
|---------|---------|-------|---------|------|
| T-XXX-01 | ... | P0/P1 | Xh | Pending |
...

---

## 任务依赖关系

```
实体设计 (已完成)
    │
    ├──> T-XXX-01 ~ T-XXX-N (可并行开发)
```

---

## T-XXX-01：{任务名称}

**功能编号**：F-XXX-01

**任务描述**：
{简要描述要实现的功能}

**前置条件**：
- [x] {实体名} 实体已设计完成

**验收标准**：
- [ ] {验收条件1}
- [ ] {验收条件2}
...

**参考**：功能设计文档 F-XXX-01

---

## 完成标准

所有任务完成后，应满足以下标准：

1. **功能完整性**
   - [ ] N 个 API 接口全部实现
   - [ ] 所有验收标准通过

2. **代码质量**
   - [ ] 代码符合 MMB 框架规范
   - [ ] 控制器方法都有对应的接口定义
   - [ ] DTO 和 RequestModel 数据注解完整

---
```

## 重要原则

1. **不要预先规定具体实现步骤**：实现步骤由 Sub Agent 根据实际情况判断

2. **不要预先规定具体的文件路径**：文件组织方式由 Sub Agent 根据框架规范判断

3. **保持任务独立性**：每个任务应该可以独立完成，不依赖其他任务

4. **验收标准明确**：每个任务必须有清晰的验收 checklist

5. **引用功能设计文档**：详细需求在功能设计文档中，任务拆解文档只引用
